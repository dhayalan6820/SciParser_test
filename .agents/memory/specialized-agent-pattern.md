---
name: Specialized-agent extension pattern (Address/Calendar/Login/Booking)
description: How to add a new deterministic per-flow agent (detector/scorer module + .agent.md spec) into Brain's existing tool loop without a new orchestration engine.
---

Each specialized agent (address, calendar, login, booking) is a self-contained module with the same shape: `has_requested_<x>`/`extract_requested_<x>` (pull the relevant confirmed_input), `detect_<x>_context` (is this obstacle-relevant UI on screen), `extract_*`/`score_*` (candidate scoring against requested values), a small dataclass tracking retries/pending-selection/spec-injected flags, and `handle_<x>_observation` functions that mutate the observation string with injected guidance and raise `ObstacleInputNeeded` after the retry budget (1-2 retries) is exhausted.

**Why:** This mirrors the CAPTCHA/OTP obstacle framework's dispatcher pattern but for silent, non-obstacle-shaped mid-run ambiguity (e.g. an address autocomplete dropdown with no exact match) — the retry-then-pause discipline prevents infinite loops while keeping detection/scoring logic testable in isolation from Brain's tool loop.

**How to apply:** Wire a new one into `brain.py` by (1) adding state init (dataclass + extracted requested-value + optional spec_guidance loaded via `spec_loader.load_spec`) right after the existing address block near the top of the run, and (2) adding a detection block in `_call_tool` immediately after the Address Agent block and before the generic `detect_obstacle` call, wrapped in try/except that re-raises `ObstacleInputNeeded` and logs other exceptions as non-fatal. Resuming after escalation reuses the ALREADY-GENERIC `pending_obstacle` resume branch — no per-category resume code needed, since obstacle categories are handled generically there. Add a `build_input_form` branch per new category in `obstacle_handler.py` before the final fallback. Gate detection on the relevant `confirmed_inputs` being present (e.g. `_requested_dates`) except for flows like booking/pagination-stall tracking that should run unconditionally on any multi-step UI.
